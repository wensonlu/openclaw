## 1. Ralph — Autonomous Coding Loop

**定位：** PRD 驱动的自主 AI 编码循环，让 AI 持续工作直到所有需求完成。

### 核心机制

```
PRD (prd.json)
  ↓
Ralph 循环启动
  ↓
选最高优先级 story（passes: false）
  ↓
创建 feature branch
  ↓
启动 AI 编码工具（Claude Code / Amp）
  ↓
跑质量检查（typecheck、tests）
  ↓
✓ 通过 → 标记 passes: true
✗ 失败 → 记录问题，下次迭代重试
  ↓
所有 story 完成 → 输出 <promise>COMPLETE</promise>
```

### 关键设计

- **干净上下文**：每次迭代是全新 AI 实例，通过 prd.json + Git 历史 + progress.txt 跨迭代记忆
- **PRD 驱动**：强制把大需求拆小，减少 AI 失控
- **Background 运行**：睡前启动，睡醒检查结果

### 文件结构

| 文件 | 作用 |
|------|------|
| `ralph.sh` | Bash 循环脚本 |
| `prd.json` | 用户故事列表 + 完成状态 |
| `progress.txt` | 追加式学习记录 |
| `skills/prd/` | PRD 生成 Skill |
| `skills/ralph/` | PRD → JSON 转换 Skill |

### 优缺点

| 优点 | 缺点 |
|------|------|
| 上手极快（copy 文件即可） | 单 Agent，无多 Agent 协作 |
| 无需搭建平台 | 纯终端，无 Web UI |
| 16K Stars 验证市场需求 | PRD 拆分质量依赖人 |

### 链接

- GitHub: github.com/snarktank/ralph
- Stars: 16,267 ⭐
- 创建时间: 2026-01-07

## 2. Archon — AI Coding Harness Builder

**定位：** 第一个开源的 AI 编程 Harness Builder，让 AI 编程变得**确定可重复**。

### 解决什么问题

当前痛点：让 AI "修一下 bug"，每次结果全靠模型心情——可能跳过规划、可能忘记跑测试、可能漏掉代码审查。

Archon 思路：用 YAML 定义工作流结构，AI 只负责每个步骤的智力输出，流程本身是确定性的。

### 类比

| 技术 | 解决的问题 |
|------|----------|
| Docker | 基础设施标准化 |
| GitHub Actions | CI/CD 标准化 |
| **Archon** | **AI 编程工作流标准化** |

### 核心机制

工作流 = 确定性节点 + AI 节点混合

```yaml
nodes:
  - id: plan          # AI: 分析需求
  - id: implement     # AI: 循环实现直到测试通过
  - id: run-tests     # 确定: bash脚本跑验证
  - id: review        # AI: 代码审查
```

### vs OpenASE

| 维度 | Archon | OpenASE |
|------|--------|---------|
| 核心抽象 | YAML 工作流 | Ticket + Workflow |
| 交互模式 | 纯异步 | 同步 + 异步 |
| 团队协作 | 单 Agent | 多 Agent 接力 |
| 成熟度 | 更成熟 | 很新（仅25天）|

### 链接

- GitHub: github.com/coleam00/Archon
- 定位: "Make AI coding deterministic and repeatable"

## 3. OpenASE — Ticket-Driven Automated SE

**定位：** Ticket 驱动的自动化软件工程平台，把工单变成可运行的代码。

### 5 步启动流程

1. **Settings** — 定义项目、状态（看板列）、连接仓库
2. **Machines** — 注册执行环境（本地/远程 WebSocket）
3. **Agents** — 注册 AI 执行者（Claude Code / Codex / Gemini CLI）
4. **Workflows** — 写 Harness + 绑定 Agent + 配置状态触发
5. **Tickets** — 创建任务，Agent 自动领取执行

### 核心概念

| 概念 | 解释 |
|------|------|
| **Ticket** | 工单 = 任务描述（做什么） |
| **Workflow** | 工作流 = 行为模板（怎么做） |
| **Harness** | Markdown 格式的执行指令，约束 Agent 行为 |
| **Skill** | 可复用技能包 |
| **Machine** | 执行环境 |
| **Project AI** | 同步交互助手，先讨论再执行 |

### 工作流程

```
User 创建 Ticket → Orchestrator 检测状态 → Agent 领取
→ 在 Machine 上执行 Workflow → Activity 全程记录 → Ticket 完成
```

### 两种执行模式

**异步（Ticket Agent）：**
- 多 Agent 协作：Design → Backend → Frontend → Testing → Review
- 每个 Ticket 跑在独立 git worktree，并行不冲突

**同步（Project AI）：**
- 需求模糊时先讨论
- 分析需求 → 起草 PRD → 初始化仓库 → 准备好再转异步

### 架构

```
Settings → Machines → Agents → Skills → Workflows → Tickets
                                                  ↓
                                              Activity
```

- GitHub: github.com/PacificStudio/openase
- Stars: 192
- 语言: Go

## 5. GuppyLM — 会说话的小鱼

**定位：** 900 万参数的 LLM，从零训练成一条会说话的鱼"Guppy"，用于 AI 教育。

### 技术规格

| 指标 | 数值 |
|------|------|
| 参数量 | 8.7M |
| 层数 | 6 |
| Hidden dim | 384 |
| Attention heads | 6 |
| 词表 | 4,096 (BPE) |
| 最大序列 | 128 tokens |
| 架构 | Vanilla Transformer |

### 训练数据

- 60K 条合成对话，覆盖 60 个主题
- 硬件：T4 GPU，约 5 分钟训练完成
- 完全跑在浏览器里（WebAssembly）

### 为什么火

| 原因 | 分析 |
|------|------|
| 教育价值极高 | 全链路拆解 LLM 训练，惠及入门者 |
| 趣味性强 | "会说话的鱼"比干燥技术项目更传播 |
| 门槛极低 | 浏览器能跑、Colab 能训练 |
| 自带数据集 | 60K 对话数据集公开在 HuggingFace |

### 链接

- GitHub: github.com/arman-bd/guppylm
- HuggingFace: huggingface.co/arman-bd/guppylm-9M
- Demo: arman-bd.github.io/guppylm
